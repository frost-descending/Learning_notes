# Ubuntu 服务器守护进程异常被杀溯源与 Systemd 托管最佳实践

## 一、 进程异常被 Kill 的溯源排查步骤

守护进程（Daemon）本该在后台稳定运行，如果突然被 kill 掉，通常不外乎两种情况：系统因资源耗尽痛下杀手（主动），或者程序自身出了致命故障/收到了外部信号（被动）。你可以按照以下由浅入深的步骤进行溯源排查：

### 1. 检查是否被系统 OOM Killer 强杀（最常见原因）

如果服务器内存不足，Linux 内核为了防止系统崩溃，会触发 **OOM (Out of Memory) Killer** 机制，挑选内存占用高的进程直接抹杀。

* **查看 OOM 历史记录：**
```bash
sudo dmesg -T | grep -i -E 'oom|kill'

```


* **查看系统相关日志：**
```bash
sudo egrep -i -r 'killed process' /var/log/

```



> **排查线索：** 如果看到类似 `Out of memory: Kill process <你的进程名> 或 <PID>` 的字样，说明是内存泄漏或瞬间内存飙升导致被内核清理了。

### 2. 查看系统综合日志 (syslog / journalctl)

如果进程是因为其他原因异常退出或收到特定信号，系统日志通常会留下蛛丝马迹。

* **普通后台进程（如 `nohup`、`setsid` 或 `&`）：**
```bash
sudo tail -n 200 /var/log/syslog

```


在日志中搜索你程序被杀的时间点，看看前后有没有其他异常事件（比如 SSH 登录、断网、其他服务崩溃等）。

### 3. 排查是否有人为或其他脚本误杀

有时候进程被杀是因为收到了 `SIGTERM (15)` 或 `SIGKILL (9)` 信号。可以检查当前用户的命令历史，或者其他自动化脚本（如定时任务 cron、部署脚本）是否误触发了 `pkill` 或 `killall`：

```bash
history | grep kill

```

### 4. 检查程序自身的日志与 Core Dump（最彻底的方案）

如果系统日志很干净，那大概率是程序内部发生了严重错误（如 C/C++ 中的**段错误 Segmentation fault**、内存越界、死锁等）而自杀。

* **检查业务日志：** 检查你程序输出重定向的文件（如 `nohup.out` 或自定义的 log 文件），重点看最后几行是否有 `panic`、`exception` 或自定义的 `error` 级别日志。
* **检查 Core Dump (核心转储)：** 如果程序因为段错误崩溃，且系统开启了 Core Dump，会生成一个进程死前的“尸体”文件，通过它可以直接还原崩溃现场。
* 查看当前系统是否允许生成 Core 文件（输出为 `0` 代表未开启，`unlimited` 代表已开启）：
```bash
ulimit -c

```


* 在 Ubuntu 中，Core 文件默认可能并不会生成在当前目录下，而是被 `apport` 服务收集了。你可以去 `/var/crash/` 目录下看看有没有近期生成的 `.crash` 文件。
* 用 GDB 追溯调用栈：
```bash
gdb ./你的程序名 /path/to/core_file
# 进入 gdb 后输入 bt 查看调用栈
(gdb) bt

```

## 二、 基础知识储备

### 1. 什么是 systemd？

`systemd` 是现代 Linux 系统（包括 Ubuntu）的**初始化系统（Init System）和服务管理器**。
当系统开机、内核加载完成后，会启动第一个用户态进程，这个进程的 PID（进程 ID）永远是 `1`。在现代 Ubuntu 中，这个 1 号进程就是 `systemd`。它是所有其他进程的“父辈”或“祖先”，可以理解为整个操作系统的“大管家”。

交给 `systemd` 管理守护进程的核心优势：

* **开机自启：** 服务器重启后，程序自动在后台跑起来。
* **崩溃自动拉起：** 如果程序突然被杀（比如因为 OOM 或段错误崩溃），`systemd` 发现后会自动重启它。
* **统一的日志管理：** 程序的所有标准输出（`stdout`）和错误输出（`stderr`）都会被自动收集。

### 2. 如何查看运行中程序的工作目录和绝对路径？

Linux 系统的 `/proc` 虚拟文件系统提供了最准确的答案。每个运行的进程在 `/proc` 下都有一个以其 PID 命名的文件夹。

#### 方法一：通过 `/proc` 查看（最底层、最准确）

假设通过 `ps axj` 查到你的程序 PID 是 `406914`，父进程 PPID 是 `1`（说明已成功脱离终端）：

```bash
sudo ls -l /proc/406914 | grep -E "cwd|exe"

```

**输出示例：**

```plaintext
lrwxrwxrwx 1 msy msy 0 May 27 13:11 cwd -> /home/msy/based-on-onethreadoneloop-web-games/OneThreadOneLoop_games/source/http
lrwxrwxrwx 1 msy msy 0 May 27 13:11 exe -> /home/msy/based-on-onethreadoneloop-web-games/OneThreadOneLoop_games/source/http/game

```

* **`cwd` (Current Working Directory)：** 指向程序当前的工作目录。决定了代码里写相对路径（如加载配置、日志、资源文件）时去哪里寻找。
* **`exe`：** 指向程序文件的绝对路径（即可执行文件本身在磁盘上的位置）。

#### 方法二：使用 `pwdx` 快捷工具

`pwdx` 命令可以直接输出某个 PID 当前的工作目录：

```bash
pwdx 406914
# 输出: 406914: /home/msy/based-on-onethreadoneloop-web-games/OneThreadOneLoop_games/source/http

```

## 三、 将 C++ 守护进程托管至 Systemd 的踩坑与实战

### 1. 编写配置文件时的“行内注释”大坑

`systemd` 的配置文件（INI 格式）**不支持**行内注释（例如 `User=msy # 运行用户`）。它会把 `#` 及其后面的内容全部当成配置的一部分解析，从而导致报 `status=217/USER` 错误或无法解析 `always` 参数。

> **正确做法：** 所有的注释 `#` 必须单独写在独立的一行。

### 2. 代码自带 `daemon()` 逻辑导致无限重启

当程序内部写了 `daemon()` 函数或自行 `fork()` 两次脱离终端时：

* **现象：** 主进程（PID A）拉起子进程后成功退出（返回 `status=0/SUCCESS`）。但 `systemd` 默认（`Type=simple`）会误认为程序已经运行结束了，由于配置了 `Restart=always`，它会热心地再次拉起程序，导致陷入“启动 -> 退出 -> 自动重启”的无限死循环。
* **解决方案：** 必须在配置中将服务类型指定为 `Type=forking`。

### 3. 完整的服务配置与部署流程

#### 步骤一：创建并编辑配置文件

```bash
sudo nano /etc/systemd/system/myserver.service

```

写入以下修正后的标准内容（注意路径与运行用户已匹配实际环境）：

```ini
# 服务描述（注意：注释必须单独成行）
[Unit]
Description=My Custom Web Server
After=network.target

[Service]
# 核心注意：因为 C++ 程序内部自带了 fork/daemon 逻辑，这里必须声明为 forking
Type=forking
# 根据实际查到的 cwd 填入工作目录
WorkingDirectory=/home/msy/based-on-onethreadoneloop-web-games/OneThreadOneLoop_games/source/http
# 根据实际查到的 exe 填入绝对路径
ExecStart=/home/msy/based-on-onethreadoneloop-web-games/OneThreadOneLoop_games/source/http/game
# 核心配置：崩溃或被 Kill 后自动重启
Restart=always
RestartSec=5
# 运行该程序的用户（ls 显示属于 msy，这里务必保持一致）
User=msy

[Install]
WantedBy=multi-user.target

```

#### 步骤二：清理旧进程并启动服务

由于旧的独立进程可能仍在占用网络端口，启动新服务前需要彻底清理：

```bash
# 1. 停止当前正在无限循环的服务
sudo systemctl stop myserver

# 2. 强行杀掉之前残余的所有独立 game 进程（防止端口冲突）
sudo pkill -9 game

# 3. 让 systemd 重新加载刚修改过的配置文件
sudo systemctl daemon-reload

# 4. 正式启动服务
sudo systemctl start myserver

# 5. 查看运行状态
sudo systemctl status myserver

```

### 4. 成功托管的验证标志

当执行 `sudo systemctl status myserver` 看到如下输出时，代表**完美托管成功**：

* 状态显示为绿色的 `Active: active (running)` 且运行时间保持稳定。
* 能够看到 `Process: xxx ... (code=exited, status=0/SUCCESS)`，说明主进程成功交接工作并退出。
* 出现稳定的 `Main PID: xxxxx (game)` 守护进程，并能看到其占用的内存（Memory）和线程数（Tasks）。

## 四、 常用服务管理命令速查

托管成功后，日常维护只需要通过以下命令即可轻松管理：

| 命令 | 作用 |
| --- | --- |
| `sudo systemctl start myserver` | 启动服务 |
| `sudo systemctl stop myserver` | 停止服务 |
| `sudo systemctl restart myserver` | 重启服务（如重新编译了程序） |
| `sudo systemctl status myserver` | 查看服务当前的详细运行状态 |
| `sudo systemctl enable myserver` | 设置为开机自动启动 |
| `sudo systemctl disable myserver` | 取消开机自启 |
| `journalctl -u myserver.service -f` | **排查神器**：实时跟踪并打印该服务的日志输出（`Ctrl+C` 退出） |
